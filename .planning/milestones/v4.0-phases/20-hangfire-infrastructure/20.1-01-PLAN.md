<objective>
Fix Hangfire dashboard redirect behavior: unauthenticated and non-Admin users navigating to
/hangfire are redirected to /Account/Login (HTTP 302) instead of receiving 401 or 403.

Purpose: Hangfire's AspNetCoreDashboardMiddleware unconditionally overwrites Response.StatusCode
to 401 or 403 after the IDashboardAuthorizationFilter returns false — discarding any redirect
that the filter attempts via Response.Redirect(). The fix inserts an app.Use() inline middleware
in Program.cs immediately before UseHangfireDashboard that intercepts /hangfire requests and
issues the 302 redirect BEFORE Hangfire's middleware can respond. AdminDashboardAuthFilter is
cleaned up to only return true/false (defense-in-depth) and no longer calls Response.Redirect().

Output: Program.cs with pre-Hangfire redirect middleware; AdminDashboardAuthFilter with redirect
calls removed. UAT tests 2 and 3 pass.
</objective>

Root cause: Hangfire's AspNetCoreDashboardMiddleware calls its own IDashboardAuthorizationFilter
chain first, then unconditionally writes 401 (unauthenticated) or 403 (non-Admin) to
Response.StatusCode AFTER the filter returns false. Response.Redirect() in the filter sets
the Location header and marks 302, but Hangfire then overwrites StatusCode with 401/403,
producing a broken response. The fix must issue the redirect BEFORE UseHangfireDashboard
runs, which means a middleware placed earlier in the pipeline.

```
if (!app.Environment.IsEnvironment("Testing"))
{
    app.Use(async (context, next) =>
    {
        if (context.Request.Path.StartsWithSegments("/hangfire"))
        {
            if (context.User.Identity?.IsAuthenticated != true)
            {
                context.Response.Redirect("/Account/Login");
                return;
            }

            if (!context.User.IsInRole("Admin"))
            {
                context.Response.Redirect("/Account/Login");
                return;
            }
        }

        await next();
    });

    app.UseHangfireDashboard("/hangfire", new DashboardOptions
    {
        Authorization = new[] { new AdminDashboardAuthFilter() }
    });
}
```

Implementation notes:
- The app.Use() call MUST be inside the !IsEnvironment("Testing") block so the Testing
  environment guard still prevents any Hangfire-related middleware from running during
  integration tests (which use the WebApplicationFactory with "Testing" environment).
- context.Request.Path.StartsWithSegments("/hangfire") is case-sensitive on Linux but the
  app runs on Windows/Docker with case-insensitive paths — this is acceptable.
- The middleware returns immediately after Response.Redirect() — it does NOT call await next()
  in the redirect branches. This ensures the response is complete before Hangfire can write
  its own status code.
- UseAuthentication and UseAuthorization already appear before this block in the pipeline
  (lines 127–128 of the current Program.cs), so context.User is fully populated with
  identity and role claims when this middleware runs.
- AdminDashboardAuthFilter remains in Authorization as defense-in-depth — it is NOT removed.

Replace the entire file content with:

```csharp
using Hangfire.Dashboard;

namespace EuphoriaInn.Service.Authorization;

public class AdminDashboardAuthFilter : IDashboardAuthorizationFilter
{
    public bool Authorize(DashboardContext context)
    {
        var httpContext = context.GetHttpContext();

        if (httpContext.User.Identity?.IsAuthenticated != true)
            return false;

        if (!httpContext.User.IsInRole("Admin"))
            return false;

        return true;
    }
}
```

Implementation notes:
- The two-step auth check is preserved: IsAuthenticated first, then IsInRole("Admin").
  This order matches the existing AdminHandler pattern in AdminHandler.cs.
- Response.Redirect() is removed from both branches. The app.Use() middleware in Program.cs
  already issued the redirect before Hangfire's middleware ran — so by the time Authorize()
  is called for a redirected user, the response is already complete and this path is never
  reached in practice. The filter is now purely a defense-in-depth gate that prevents
  Hangfire from serving dashboard content if somehow the pre-middleware is bypassed.
- No new using directives are needed. The existing "using Hangfire.Dashboard;" is sufficient.
