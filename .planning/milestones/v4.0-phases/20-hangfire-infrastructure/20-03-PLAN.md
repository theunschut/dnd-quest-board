**Change 2 — Add Hangfire service registration** (insert after the `builder.Services.AddDomainServices(builder.Configuration);` block):

```csharp
if (!builder.Environment.IsEnvironment("Testing"))
{
    builder.Services.AddHangfire(config => config
        .SetDataCompatibilityLevel(CompatibilityLevel.Version_180)
        .UseSimpleAssemblyNameTypeSerializer()
        .UseRecommendedSerializerSettings()
        .UseSqlServerStorage(
            builder.Configuration.GetConnectionString("DefaultConnection"),
            new SqlServerStorageOptions
            {
                CommandBatchMaxTimeout = TimeSpan.FromMinutes(5),
                SlidingInvisibilityTimeout = TimeSpan.FromMinutes(5),
                QueuePollInterval = TimeSpan.Zero,
                UseRecommendedIsolationLevel = true,
                DisableGlobalLocks = true
            }));

    builder.Services.AddHangfireServer(options =>
    {
        options.WorkerCount = 2;
    });
}
```

CRITICAL: Both `AddHangfire` AND `AddHangfireServer` are wrapped in the `!IsEnvironment("Testing")`
guard. This prevents Hangfire's SqlServerStorage from attempting to connect to / create schema
against a SQL Server that does not exist in the integration test environment (which uses SQLite).
See RESEARCH.md Pitfall 2 and Open Question 1 for rationale.

**Change 3 — Add UseHangfireDashboard middleware** (insert immediately after `app.UseAuthorization();`,
before `app.MapControllerRoute(...)`):

```csharp
app.UseHangfireDashboard("/hangfire", new DashboardOptions
{
    Authorization = new[] { new AdminDashboardAuthFilter() }
});
```

This placement satisfies D-03: UseHangfireDashboard MUST appear after UseAuthentication and
UseAuthorization so that `context.GetHttpContext().User` has claims populated when the filter runs.
It must also appear BEFORE `app.MapControllerRoute` to avoid the "response already started" pitfall
(RESEARCH.md Pitfall 5).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → UseHangfireDashboard middleware | Requests to /hangfire cross into the Hangfire middleware pipeline; AdminDashboardAuthFilter is the gatekeeper |
| Hangfire worker → DI container | Background job execution resolves services from the DI container; Testing guard prevents this boundary from activating in test runs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-05 | Elevation of Privilege | UseHangfireDashboard registration order | mitigate | `UseHangfireDashboard` is placed after `UseAuthentication` and `UseAuthorization` in Program.cs (per D-03); if placed before, `httpContext.User` would always be unauthenticated — this ordering is verifiable by code review of Program.cs |
| T-20-06 | Elevation of Privilege | Response.Redirect after response started | mitigate | `UseHangfireDashboard` is placed before `app.MapControllerRoute` — prevents "response already started" condition (RESEARCH.md Pitfall 5) that would cause 500 instead of redirect |
| T-20-07 | Denial of Service | Hangfire server in Testing environment | mitigate | Both `AddHangfire` and `AddHangfireServer` are guarded by `!IsEnvironment("Testing")`; prevents SqlServerStorage from attempting schema creation against the SQLite test DB, which would throw SqlException and break all 134 tests |
| T-20-08 | Information Disclosure | Hangfire job argument visibility in dashboard | accept | Job arguments (e.g., email addresses in future Phase 21–22 jobs) are visible in the Hangfire dashboard; this is acceptable because the dashboard is Admin-role-gated; no PII is exposed to non-Admin users |
</threat_model>
