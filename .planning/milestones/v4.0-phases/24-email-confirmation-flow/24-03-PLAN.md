Call `await emailService.SendAsync(user.Email!, "Confirm your D&D Quest Board account", html);`. Set `TempData["Success"] = $"Confirmation email sent to {user.Name}.";` and `return RedirectToAction(nameof(Users));`. Use TempData keys `Success`/`Error` exactly (not `SuccessMessage` — that existing inconsistency in ResetPassword must not be copied here).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin browser → POST /Admin/SendConfirmationEmail | Untrusted form input (`userId`) crosses into the controller |
| controller → email client (callback URL) | The generated link travels through an external email channel before returning as a GET to the app |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Tampering / CSRF | SendConfirmationEmail POST | mitigate | `[ValidateAntiForgeryToken]` on the action; the view form posts the antiforgery token. Action also guarded by class-level `[Authorize(Policy = "AdminOnly")]` on AdminController. |
| T-24-02 | Access Control | SendConfirmationEmail | mitigate | Inherited `[Authorize(Policy = "AdminOnly")]` on AdminController restricts dispatch to admins (admin has already verified the user exists). |
| T-24-03 | Input Validation | `userId` parameter | mitigate | `userService.GetByIdAsync(userId)` null-check returns to Users without throwing; null/empty email also short-circuits with a TempData error. |
| T-24-08 | Information Disclosure | Relative callback URL leaking to wrong domain | mitigate | `Url.Action(..., Request.Scheme)` forces an absolute URL bound to the request host — never a relative path that resolves against the email client's domain. |
| T-24-SC | Tampering | npm/pip/cargo installs | accept | No package installs — `WebEncoders` ships with the ASP.NET Core framework; `IEmailService`/`IIdentityService` already registered. No audit table required. |
</threat_model>
