## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server (User-Agent header) | Untrusted UA selects which layout renders; no access-control impact |
| Razor layout → Identity authorization | `_Layout.Mobile.cshtml` nav items gated by `AuthorizeAsync` policy checks, same as desktop |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-04 | Elevation of Privilege | `_Layout.Mobile.cshtml` offcanvas nav | mitigate | The mobile nav MUST use the IDENTICAL `AuthorizeAsync(User, "AdminOnly")` and `AuthorizeAsync(User, "DungeonMasterOnly")` checks as `_Layout.cshtml` (D-01). Nav visibility is cosmetic — the underlying controller actions retain their `[Authorize]`/policy attributes — but mirroring the checks prevents showing privileged links to unauthorized users. Verified by acceptance criterion requiring identical policy names. |
| T-12-05 | Spoofing | User-Agent → mobile layout | accept | Spoofing a mobile UA only changes the rendered shell; the mobile layout serves the same routes, same data, and same authorization as desktop. No security boundary is crossed. ASVS L1: not security-relevant. |
| T-12-06 | Information Disclosure | `_ViewStart.cshtml` reading `Context.Items["IsMobile"]` | accept | Boolean layout flag only; the null-safe `is true` pattern prevents an exception path that could leak a stack trace on non-MVC requests. No sensitive data in the flag. |
| T-12-SC | Tampering | npm/nuget installs | mitigate | No packages installed this phase (RESEARCH.md Package Legitimacy Audit: zero new dependencies). No install checkpoint required. |
</threat_model>
