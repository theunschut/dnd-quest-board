## Color

Existing Bootstrap semantic palette — do not introduce custom hex values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Bootstrap `body` bg (`#fff` / dark body text) | Page background, card body |
| Secondary (30%) | Bootstrap `bg-dark` (`#212529`) | Nav bar, card header (`modern-card-header`) |

This is already used in the Actions column for the Edit button (`btn-info`), making it the correct semantic choice for a non-destructive utility action that is distinct from role-management actions (which use `btn-warning`/`btn-danger`).

## Component Inventory

### 1. Admin/Users — "Send Confirmation Email" row button

**Location:** `Views/Admin/Users.cshtml`, Actions column (`td class="text-end"`)

**Visibility rule:** Rendered only when `userModel.User.EmailConfirmed == false`. Absent (no placeholder, no disabled state) for confirmed users.

**Markup contract:**

```html
@if (!userModel.User.EmailConfirmed)
{
    <form asp-action="SendConfirmationEmail" method="post" class="d-inline me-2">
        <input type="hidden" name="userId" value="@userModel.User.Id" />
        <button type="submit" class="btn btn-sm btn-info">
            <i class="fas fa-envelope me-1"></i>
            Send Confirmation Email
        </button>
    </form>
}
```

**Position in Actions column:** After Promote/Demote buttons, before the Edit button. Final order: [Promote/Demote] → [Send Confirmation Email (if unconfirmed)] → [Edit] → [Delete].

**Source decision:** D-04 (CONTEXT.md). Pattern mirrors existing `PromoteToAdmin`/`DemoteToPlayer` forms (`d-inline me-2`, `btn-sm`, `input type="hidden" name="userId"`).

---

### 2. Admin/Users — TempData feedback banner

**Location:** `Views/Admin/Users.cshtml`, top of `card-body modern-card-body` div, before the table.

The Users page currently has no TempData banner block. It must be added. Pattern taken from `Quest/Manage.cshtml`:

```html
@if (TempData["Success"] != null)
{
    <div class="alert alert-success alert-dismissible fade show" role="alert">
        <i class="fas fa-check-circle me-2"></i>
        @TempData["Success"]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
}

@if (TempData["Error"] != null)
{
    <div class="alert alert-danger alert-dismissible fade show" role="alert">
        <i class="fas fa-exclamation-triangle me-2"></i>
        @TempData["Error"]
        <button type="button" class="btn-close" data-bs-dismiss="alert" aria-label="Close"></button>
    </div>
}
```

**TempData key convention:** `TempData["Success"]` and `TempData["Error"]` — matching `Quest/Manage.cshtml` and `Shop/Details.cshtml`. Note: `AdminController.ResetPassword` currently uses `TempData["SuccessMessage"]` — this is an existing inconsistency. Phase 24 must use `TempData["Success"]` / `TempData["Error"]` to match the view keys above.

**Source decision:** D-05 (CONTEXT.md).

## Interaction States

### "Send Confirmation Email" button

| State | Behavior |
|-------|----------|
| Normal | `btn-sm btn-info`, enabled |
| Already confirmed | Button absent — not rendered at all |
| POST success | Page reloads; `TempData["Success"]` banner appears at top of Users page |
| POST error (user not found / send failure) | Page reloads; `TempData["Error"]` banner appears at top of Users page |

No loading state, no optimistic UI — standard synchronous POST-redirect-GET.

### ConfirmEmail callback (`GET /Account/ConfirmEmail`)

| State | Behavior |
|-------|----------|
| Valid token | `UserManager.ConfirmEmailAsync` succeeds → `RedirectToAction(nameof(Login))` with `TempData["Success"]` |
| Invalid / expired token | `ConfirmEmailAsync` returns failure → `RedirectToAction(nameof(Login))` with `TempData["Error"]` |
| Missing userId or token params | `RedirectToAction(nameof(Login))` with `TempData["Error"]` |

No dedicated error page. All paths redirect to Login. Source: D-03.

## ViewModel Contract

`UserManagementViewModel` must gain `bool EmailConfirmed` to drive button visibility.

```csharp
public bool EmailConfirmed { get; set; }
```

Populated in `AdminController.Users` from `userModel.User.EmailConfirmed` (once `EmailConfirmed` is on the `User` domain model per D-01).

View condition: `@if (!userModel.EmailConfirmed)` — not `@if (!userModel.User.EmailConfirmed)` — keeps view logic on the ViewModel rather than reaching into the nested model.
