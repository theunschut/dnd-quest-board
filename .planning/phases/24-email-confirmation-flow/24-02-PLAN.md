<objective>
Extend the existing `IIdentityService` / `IdentityService` abstraction with two methods that wrap ASP.NET Identity's email-confirmation token APIs: `GenerateEmailConfirmationAsync(int userId)` returning the raw token, and `ConfirmEmailAsync(int userId, string token)` returning an `IdentityResult`.

Purpose: `UserManager<UserEntity>` is only accessible in the Repository layer. Controllers (plans 03/04) must route token generation and confirmation through `IIdentityService` exactly as `AdminResetPasswordAsync` does — never injecting `UserManager` directly (layer boundary). These two methods are consumed by `AdminController.SendConfirmationEmail` (plan 03) and `AccountController.ConfirmEmail` (plan 04 — note: the callback action ships in plan 04's wave, but the interface it calls must exist first; this plan is its dependency).
Output: Two new interface members and their implementations, mirroring the established `AdminResetPasswordAsync` pattern.
</objective>

    In `EuphoriaInn.Repository/IdentityService.cs`, implement both. `GenerateEmailConfirmationAsync`: call `await userManager.FindByIdAsync(userId.ToString())`; if the entity is null return `null`; otherwise return `await userManager.GenerateEmailConfirmationTokenAsync(entity)`. `ConfirmEmailAsync`: call `await userManager.FindByIdAsync(userId.ToString())`; if the entity is null return `IdentityResult.Failed(new IdentityError { Description = "User not found." })`; otherwise return `await userManager.ConfirmEmailAsync(entity, token)`. Do NOT do any token encoding/decoding inside the service — encoding is the controller's responsibility (plans 03/04 use `WebEncoders.Base64UrlEncode`/`Base64UrlDecode`). ## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-06 | Spoofing / Cryptography | Confirmation token generation | mitigate | Use `UserManager.GenerateEmailConfirmationTokenAsync` (Identity-issued, security-stamp bound, self-expiring) — do NOT hand-roll a GUID/HMAC token. |
| T-24-07 | Repudiation | Token replay | mitigate | `UserManager.ConfirmEmailAsync` is single-use: confirming rotates the security stamp, invalidating the token for re-use. No custom replay logic needed. |
| T-24-05 | Input Validation | Non-existent userId | mitigate | `FindByIdAsync` null-check returns `null` token / `IdentityResult.Failed` — never throws on unknown userId. |
| T-24-SC | Tampering | npm/pip/cargo installs | accept | No package installs — `UserManager<UserEntity>` and `IdentityResult` are already referenced in IdentityService.cs. No audit table required. |
