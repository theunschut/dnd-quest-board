# Phase 14 — UI Design Contract

> Visual and interaction contract for Phase 14: Calendar.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase extends Phase 13's design contract. All baseline tokens (spacing, typography, color, touch targets) are **inherited** from Phase 13-UI-SPEC.md and mobile.css. This document specifies only what Phase 14 adds or overrides.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Bootstrap 5.3.0 (CDN) — utility classes only |
| Icon library | Font Awesome 6.4.0 (CDN) |
| Font | Google Fonts — Cinzel (headings); system sans-serif (body) |

No shadcn. No Radix. No component registry. All UI is Bootstrap 5 utility classes + custom CSS in per-page files.

Source: Phase 13 UI-SPEC.md Design System (inherited verbatim); `_Layout.Mobile.cshtml` lines 8–11.

## Spacing Scale

Inherited from Phase 13 UI-SPEC.md. No changes.

| Token | Value | Bootstrap Util | Usage |
|-------|-------|---------------|-------|
| xs | 4px | `gap-1`, `me-1` | Icon-to-text gaps |
| sm | 8px | `p-2`, `gap-2`, `mb-2` | Button row internal gap, section vertical gap |
| md | 16px | `p-3`, `mb-3`, `gap-3` | Card body padding, between-section gap |
| lg | 24px | `p-4`, `mb-4` | Section breaks |
| xl | 32px | `mb-5` | Page-level top padding |

Phase 14 exceptions:
- Month nav chevron buttons: `min-height: 44px; min-width: 44px` — enforced by mobile.css `.btn` rule, no override needed.
- Vote button row gap: `gap-2` (8px) between the three equal-width buttons — existing token.
- Day label top padding: `pt-3` (12px from bottom of previous quest entry to next day label) — see agenda layout.

Source: 14-CONTEXT.md D-15; Phase 13 UI-SPEC.md Spacing Scale.

## Typography

Inherited from Phase 13 UI-SPEC.md. Four sizes, two weights only.

| Role | Size | Weight | Line Height | Font | Bootstrap Class |
|------|------|--------|-------------|------|----------------|
| Body | 16px | 400 | 1.5 | System sans-serif | — (mobile.css baseline) |
| Label / meta | ~14px (0.875em) | 400 | 1.4 | System sans-serif | `<small>` or `.small` |
| Card title / section heading | 20px | 700 | 1.2 | System sans-serif | `h5` or `fw-bold fs-5` |
| Page heading | 24px | 700 | 1.2 | Cinzel, serif | `h4` + Cinzel |

Phase 14 additions:
- **Day label** (agenda section header): 14px, weight 700, letter-spacing 0.08em, all-uppercase. Rendered as `<div class="agenda-day-label text-uppercase fw-bold small">`. Color inherits `color: #1a0f08` from mobile.css `h1-h6` rule via explicit inline color or `.agenda-day-label` rule in `calendar.mobile.css`.
- **Month name heading** (nav bar center): 20px, weight 700, Cinzel font. Use `<span class="fw-bold cinzel-heading">@Model.MonthName</span>`.
- **Quest time meta**: 14px (0.875em via `<small>`), weight 400. Same as existing meta text pattern.

Note: Day label uppercase is achieved by Bootstrap `.text-uppercase` — do not hard-code `text-transform: uppercase` in CSS (Bootstrap util is sufficient).

Source: 14-CONTEXT.md D-10, Specifics; Phase 13 UI-SPEC.md Typography.

## View-by-View Visual Contract

### View 1: Calendar/Index.Mobile.cshtml (CAL-01 — CAL-04)

**Purpose:** Replace the 7-column desktop grid calendar with a mobile-first agenda list. Navigates month-by-month. Only days with quests render (D-09).

**CSS loaded via:** `@section Styles { <link href="~/css/calendar.mobile.css" asp-append-version="true" rel="stylesheet" /> }`

**Layout structure (top to bottom):**
```
<div class="container-fluid px-2 mt-2">
  [Month nav bar]
  [Agenda list — one section per day-with-quests]
  [Empty state — when month has no quests]
</div>
```

**Month navigation bar (D-01, D-02):**
```html
<div class="d-flex justify-content-between align-items-center mb-3 calendar-month-nav">
    <a href="@Url.Action("Index", new { year = prevYear, month = prevMonth })"
       class="btn btn-primary calendar-nav-btn" aria-label="Previous month">
        <i class="fas fa-chevron-left"></i>
    </a>
    <span class="fw-bold cinzel-heading calendar-month-title">
        @Model.MonthName
    </span>
    <a href="@Url.Action("Index", new { year = nextYear, month = nextMonth })"
       class="btn btn-primary calendar-nav-btn" aria-label="Next month">
        <i class="fas fa-chevron-right"></i>
    </a>
</div>
```
- Chevron buttons: `min-height: 44px; min-width: 44px` (mobile.css `.btn` sets min-height; `calendar.mobile.css` adds `min-width: 44px` and `width: 44px` to `.calendar-nav-btn`)
- Month name: centered via flexbox `justify-content-between`; Cinzel font; 20px weight 700
- Full-width row: occupies `container-fluid` width

**Agenda list (D-09, D-10, D-11, D-12):**

Each day with at least one quest renders one section:

```html
@foreach (var day in Model.GetCalendarDays().Where(d => !d.IsEmpty && d.QuestsOnDay.Any()))
{
    <div class="agenda-day-section mb-2">
        <!-- Day label: SATURDAY, JUNE 14 -->
        <div class="agenda-day-label text-uppercase fw-bold small mb-1">
            @day.Date.ToString("dddd, MMMM d").ToUpper()
        </div>
        <!-- Quest entries for this day -->
        @foreach (var questOnDay in day.QuestsOnDay)
        {
            <div class="agenda-quest-entry @(questOnDay.IsFinalized ? "agenda-quest-finalized" : "agenda-quest-proposed")"
                 onclick="window.location.href='@Url.Action("Details", "Quest", new { id = questOnDay.Quest.Id })'">
                <div class="d-flex justify-content-between align-items-center">
                    <span class="agenda-quest-title fw-bold">@questOnDay.Quest.Title</span>
                    <small class="text-muted ms-2 flex-shrink-0">
                        @questOnDay.ProposedDate.Date.ToString("HH:mm")
                    </small>
                </div>
            </div>
        }
    </div>
}
```

Notes:
- Day label format: `day.Date.ToString("dddd, MMMM d").ToUpper()` — produces "SATURDAY, JUNE 14" (CAL-02 exact format from D-10)
- `day.QuestsOnDay` is NOT capped at 3 (unlike the desktop `Take(3)`) — all quests per day render (D-09 / CONTEXT.md Code Insights)
- Tap navigation: `onclick="window.location.href='...'"` — established codebase pattern (CAL-04)
- Warning icon (no building key): omitted from agenda entries per Claude's Discretion. The agenda is read-only navigation; the DM profile / quest details surface this information. Keeps the list clean.

**Empty state (month with no quests — Claude's Discretion):**
```html
<div class="text-center text-muted py-5">
    <i class="fas fa-calendar-alt fa-3x mb-3"></i>
    <h5>No Quests This Month</h5>
    <p>No adventures are planned for @Model.MonthName. Check another month.</p>
</div>
```
Copy: heading "No Quests This Month", body "No adventures are planned for [month name]. Check another month."

### View 2: _Calendar.Mobile.cshtml (D-03 — D-08, CAL-05)

**Purpose:** Shared partial replacing the desktop 7-column grid `_Calendar.cshtml`. Renders a per-date vertical list with `[✓ Yes] [✗ No] [? Maybe]` button rows. Serves both "Choose a Date" (non-signed-up, initial vote) and "Update Your Vote" (signed-up player, update vote) flows.

**CSS scope:** `_Calendar.Mobile.cshtml` is a shared partial — it has no per-page CSS of its own. Its host views supply the CSS:
- `Details.Mobile.cshtml` already loads `quests.mobile.css` (existing).
- Calendar-partial-specific mobile rules for the vote button row layout are added to `quests.mobile.css` under the selector `.calendar-partial-mobile` to keep classes predictable. No new CSS file is needed for the partial itself.

**Vote button layout (D-03, D-04, D-05):**

Each proposed date renders one entry consisting of:
1. Date label: day name + date + time (e.g. "Saturday, June 14 at 20:00")
2. Three-button horizontal row: `[✓ Yes] [✗ No] [? Maybe]` — equal-width, 44px min-height

```html
@foreach (var day in Model.GetCalendarDays())
{
    @foreach (var questOnDay in day.QuestsOnDay.Where(q => isDetailsPage && q.Quest.Id == currentQuestId))
    {
        <div class="calendar-date-entry-mobile mb-3">
            <!-- Date label + quest name -->
            <div class="calendar-date-label-mobile mb-2">
                <span class="fw-bold">@questOnDay.ProposedDate.Date.ToString("dddd, MMMM d")</span>
                <span class="text-muted ms-1">at @questOnDay.ProposedDate.Date.ToString("HH:mm")</span>
            </div>
            <!-- Vote buttons (initial vote flow — non-signed-up) -->
            @if (!questOnDay.IsFinalized && !(bool)isPlayerSignedUp)
            {
                @{
                    var voteIndexLookup = ViewBag.VoteIndexLookup as Dictionary<int, int>;
                    var voteIndex = voteIndexLookup?.ContainsKey(questOnDay.ProposedDate.Id) == true
                        ? voteIndexLookup[questOnDay.ProposedDate.Id] : -1;
                }
                @if (voteIndex >= 0)
                {
                    <div class="d-flex gap-2 calendar-vote-row">
                        <input type="radio" class="btn-check" name="DateVotes[@voteIndex].Vote"
                               id="vote_@(voteIndex)_yes" value="2" autocomplete="off">
                        <label class="btn btn-outline-success flex-fill" for="vote_@(voteIndex)_yes">
                            <i class="fas fa-check me-1"></i>Yes
                        </label>

                        <input type="radio" class="btn-check" name="DateVotes[@voteIndex].Vote"
                               id="vote_@(voteIndex)_no" value="0" autocomplete="off">
                        <label class="btn btn-outline-danger flex-fill" for="vote_@(voteIndex)_no">
                            <i class="fas fa-times me-1"></i>No
                        </label>

                        <input type="radio" class="btn-check" name="DateVotes[@voteIndex].Vote"
                               id="vote_@(voteIndex)_maybe" value="1" autocomplete="off">
                        <label class="btn btn-outline-warning flex-fill" for="vote_@(voteIndex)_maybe">
                            <i class="fas fa-question me-1"></i>Maybe
                        </label>
                    </div>
                }
            }
            <!-- Vote buttons (update vote flow — signed-up player) -->
            @else if ((bool)isPlayerSignedUp && isUpdateMode)
            {
                @{
                    var updateVoteIndex = updateVoteIndexLookup?.ContainsKey(questOnDay.ProposedDate.Id) == true
                        ? updateVoteIndexLookup[questOnDay.ProposedDate.Id] : -1;
                }
                @if (updateVoteIndex >= 0)
                {
                    <div class="d-flex gap-2 calendar-vote-row">
                        <input type="radio" class="btn-check" name="DateVotes[@updateVoteIndex].Vote"
                               id="update_vote_@(updateVoteIndex)_yes" value="2" autocomplete="off"
                               @(isCheckedYes ? "checked" : "")>
                        <label class="btn btn-outline-success flex-fill" for="update_vote_@(updateVoteIndex)_yes">
                            <i class="fas fa-check me-1"></i>Yes
                        </label>

                        <input type="radio" class="btn-check" name="DateVotes[@updateVoteIndex].Vote"
                               id="update_vote_@(updateVoteIndex)_no" value="0" autocomplete="off"
                               @(isCheckedNo ? "checked" : "")>
                        <label class="btn btn-outline-danger flex-fill" for="update_vote_@(updateVoteIndex)_no">
                            <i class="fas fa-times me-1"></i>No
                        </label>

                        <input type="radio" class="btn-check" name="DateVotes[@updateVoteIndex].Vote"
                               id="update_vote_@(updateVoteIndex)_maybe" value="1" autocomplete="off"
                               @(isCheckedMaybe ? "checked" : "")>
                        <label class="btn btn-outline-warning flex-fill" for="update_vote_@(updateVoteIndex)_maybe">
                            <i class="fas fa-question me-1"></i>Maybe
                        </label>
                    </div>
                }
            }
        </div>
    }
}
```

**Vote button order (D-03):** `[✓ Yes] [✗ No] [? Maybe]` left to right. Matches the icon+text from the desktop partial (check/times/question-mark).

**Active state (D-05):** The pre-selected radio `checked` attribute highlights the active button via Bootstrap's `.btn-check:checked + .btn-outline-*` CSS. No JavaScript needed — this is native Bootstrap behavior. The `calendar.css` `.vote-buttons .btn-check:checked` rules do not apply on mobile (those selectors are desktop-only and mobile.css does not load `calendar.css`). Bootstrap 5.3.0 CDN provides the standard `.btn-check:checked + .btn-outline-success` fill behavior.

**Update Your Vote section in Details.Mobile.cshtml (D-06, D-07, D-08):**

The existing Phase 13 three-button block (lines 79–97 of `Details.Mobile.cshtml`) is **replaced** with the unified partial call:

```html
@if (User.Identity?.IsAuthenticated == true && isPlayerSignedUp && Model.Quest?.IsFinalized == false)
{
    <div class="quest-section-card-mobile mb-3">
        <h6 class="quest-section-heading mb-3">
            <i class="fas fa-vote-yea me-1"></i>Update Your Vote
        </h6>
        @foreach (var calendarMonth in ViewBag.CalendarMonths as List<CalendarViewModel> ?? new List<CalendarViewModel>())
        {
            @await Html.PartialAsync("_Calendar", calendarMonth)
        }
    </div>
}
```

The section heading "Update Your Vote" stays as-is (D-08). The `MobileViewLocationExpander` auto-serves `_Calendar.Mobile.cshtml` instead of `_Calendar.cshtml` on mobile.

## Interaction Contract

### Tap Navigation (CAL-04)

Agenda quest entries navigate via `onclick="window.location.href='@Url.Action("Details", "Quest", new { id = questOnDay.Quest.Id })'"` — established codebase pattern from Phase 13.

Touch target: entire `.agenda-quest-entry` card is tappable (not just the title text). `cursor: pointer` set in `calendar.mobile.css`.

### Month Navigation (D-01)

Month nav uses `<a href="...">` (standard anchor links, not AJAX). Navigation uses the same URL pattern as `Index.cshtml`:
- Previous: `Url.Action("Index", new { year = Model.FirstDayOfMonth.AddMonths(-1).Year, month = Model.FirstDayOfMonth.AddMonths(-1).Month })`
- Next: `Url.Action("Index", new { year = Model.FirstDayOfMonth.AddMonths(1).Year, month = Model.FirstDayOfMonth.AddMonths(1).Month })`

### Vote Form Submission (_Calendar.Mobile.cshtml)

Votes are radio buttons inside the existing form on `Details.Mobile.cshtml` — same form POST pattern as the desktop `_Calendar.cshtml`. No new JavaScript is introduced by `_Calendar.Mobile.cshtml`. The parent view (`Details.Mobile.cshtml`) already has the `<form>` wrapper and submit button.

ViewBag/ViewData flow (D-07):
- `ViewBag.IsDetailsPage` — `true` when rendered from Details; `false` from Calendar Index
- `ViewBag.CurrentQuestId` — `int?` for filtering to the current quest's dates
- `ViewBag.VoteIndexLookup` — `Dictionary<int, int>` for initial vote form binding
- `ViewData["UpdateVoteIndexLookup"]` — `Dictionary<int, int>` for update vote form binding
- `ViewBag.IsPlayerSignedUp` — `bool` routing between vote flows
- `ViewBag.CurrentUserId` — `int?` for pre-selecting the current vote state

### No New JS Patterns

Phase 14 introduces no new JavaScript patterns. `_Calendar.Mobile.cshtml` uses only:
- Standard HTML radio buttons (`btn-check` pattern from Bootstrap 5)
- Bootstrap's native `.btn-check:checked` CSS for active state highlighting

`Calendar/Index.Mobile.cshtml` declares:
```cshtml
@section Styles {
    <link href="~/css/calendar.mobile.css" asp-append-version="true" rel="stylesheet" />
```
