# Phase 20 Plan 20.1-01: Hangfire Dashboard Redirect Fix Summary

**One-liner:** Pre-Hangfire inline middleware in Program.cs issues HTTP 302 to /Account/Login before Hangfire's AspNetCoreDashboardMiddleware can overwrite the status code with 401/403.

## What Was Built

Fixed the Hangfire dashboard redirect behavior so unauthenticated and non-Admin users receive a proper HTTP 302 redirect to /Account/Login rather than a 401 or 403 error.

**Root cause addressed:** Hangfire's `AspNetCoreDashboardMiddleware` unconditionally overwrites `Response.StatusCode` to 401 or 403 after `IDashboardAuthorizationFilter` returns false — discarding any redirect that the filter attempted via `Response.Redirect()`. The fix inserts an `app.Use()` inline middleware in `Program.cs` immediately before `UseHangfireDashboard`, intercepting `/hangfire` requests and issuing the 302 redirect before Hangfire's middleware runs.

## Key Changes

### Program.cs
Added `app.Use()` inline middleware inside the `!IsEnvironment("Testing")` block, immediately before `app.UseHangfireDashboard`:
- Checks `StartsWithSegments("/hangfire")` before acting
- Redirects unauthenticated users (`IsAuthenticated != true`) to `/Account/Login`
- Redirects authenticated non-Admin users (`!IsInRole("Admin")`) to `/Account/Login`
- Calls `await next()` only for authorized Admin users
- Placed inside `!IsEnvironment("Testing")` block: Testing environment guard still prevents any Hangfire middleware from running during integration tests

### AdminDashboardAuthFilter.cs
Removed both `Response.Redirect("/Account/Login")` calls. The filter now:
- Returns `false` for unauthenticated users
- Returns `false` for non-Admin users
- Returns `true` for authenticated Admin users
- Acts as defense-in-depth gate only — redirect responsibility belongs to the `app.Use()` middleware
