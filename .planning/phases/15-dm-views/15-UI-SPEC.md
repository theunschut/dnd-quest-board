# Phase 15 — UI Design Contract

> Visual and interaction contract for Phase 15: DM Views.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase extends Phase 13's and Phase 14's design contracts. All baseline tokens (spacing,
typography, color, touch targets) are **inherited** from Phase 13-UI-SPEC.md and mobile.css.
This document specifies only what Phase 15 adds or overrides, plus the full view-by-view
contract for the three new mobile views.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Bootstrap 5.3.0 (CDN) — utility classes only |
| Icon library | Font Awesome 6.4.0 (CDN) |
| Font | Google Fonts — Cinzel (headings); system sans-serif (body) |

No shadcn. No Radix. No component registry. All UI is Bootstrap 5 utility classes + custom CSS
in per-page files.

Source: Phase 13 UI-SPEC.md Design System (inherited verbatim); `_Layout.Mobile.cshtml` lines 8–11.

## Spacing Scale

Inherited from Phase 13 UI-SPEC.md. No changes.

| Token | Value | Bootstrap Util | Usage |
|-------|-------|---------------|-------|
| xs | 4px | `gap-1`, `me-1`, `ms-1` | Icon-to-text gaps, badge internal padding |
| sm | 8px | `p-2`, `py-2`, `gap-2`, `mb-2` | Card internal padding, badge margin, button row gap |
| md | 16px | `p-3`, `px-3`, `mb-3`, `gap-3` | Card body padding, between-section gaps |
| lg | 24px | `p-4`, `mb-4`, `gap-4` | Card stack gap, section breaks |
| xl | 32px | `mb-5` | Page-level top padding above card |
| 2xl | 48px | — | Not used in mobile views |
| 3xl | 64px | — | Not used in mobile views |

Phase 15 exceptions:
- Form fields: `mb-3` (16px) between each field — inherited from Bootstrap `.mb-3` on form groups,
  consistent with desktop Create.cshtml structure.
- Input-group proposed date rows: `mb-3` between date rows, `gap-2` (8px) inside the input-group
  between `datetime-local` input and Remove button (Bootstrap `input-group` default spacing).
- Glass card padding: `padding: 12px` (not a Bootstrap util — declared in per-page CSS, consistent
  with `.quest-section-card-mobile` pattern from quests.mobile.css).
- Touch targets: 44px minimum height on all interactive elements — set globally in mobile.css.
  Applies to `datetime-local` inputs, checkboxes' clickable labels, Finalize button, form submit.

Source: 15-CONTEXT.md D-01, D-02, D-10; Phase 13 UI-SPEC.md Spacing Scale.

## Typography

Inherited from Phase 13 UI-SPEC.md. Four sizes, two weights only.

| Role | Size | Weight | Line Height | Font | Bootstrap Class |
|------|------|--------|-------------|------|----------------|
| Body | 16px | 400 | 1.5 | System sans-serif | — (mobile.css baseline) |
| Label / meta | ~14px (0.875em) | 400 | 1.4 | System sans-serif | `<small>` or `.small` |
| Card title / section heading | 20px | 700 | 1.2 | System sans-serif | `h5` or `fw-bold fs-5` |
| Page heading | 24px | 700 | 1.2 | Cinzel, serif | `h4` + Cinzel |

Phase 15 additions:
- **Form labels**: 16px, weight 400 — standard Bootstrap `.form-label` inherits body. No override.
- **Section sub-headings** (e.g. "Players", "Assistant DMs", "Spectators"): `h6` rendered as 16px
  weight 700 via Bootstrap default `h6` — no custom font-size declaration needed.
- **Vote count badges** (condensed format): `font-size: 0.75rem` (12px), weight 400 via
  `.badge` Bootstrap default. Text-shadow disabled on badges per Phase 13 pattern.
- **Bio text on DM Profile**: 16px, weight 400, `white-space: pre-wrap` — matches desktop Profile.cshtml
  `<p style="white-space: pre-wrap;">`. No custom CSS declaration needed; body inherits from mobile.css.

Source: Phase 13 UI-SPEC.md Typography; 15-CONTEXT.md D-04, D-07.

### Vote Count Badge Color Contract (D-04 — locked)

Condensed format: `3 Yes · 1 Maybe · 0 No` — three inline `<span class="badge">` elements on the
date heading row.

| Vote type | Badge class | Icon | Text format |
|-----------|-------------|------|-------------|
| Yes | `bg-success` | `fas fa-check me-1` | `N Yes` |
| Maybe | `bg-warning text-dark` | `fas fa-question me-1` | `N Maybe` |
| No | `bg-danger` | `fas fa-times me-1` | `N No` |

Separator between badges: `·` (middle dot, `&middot;`) rendered as plain text between badge spans.
No separator class needed; natural spacing from `gap-1` on the badge container div.

Source: 15-CONTEXT.md D-04, Specifics.

Accent reserved for: Quest Create submit CTA ("Create Quest") and "Add Another Date Option" only.
Primary brown gradient reserved for: Quest Manage "Finalize Quest" button.

## View-by-View Visual Contract

### View 1: Quest/Create.Mobile.cshtml (DMVIEW-01)

**Requirements satisfied:** DMVIEW-01 — single-column, touch-friendly, vertical scroll only.
**CSS loaded via:** `@section Styles { <link href="~/css/dm-create.mobile.css" asp-append-version="true" rel="stylesheet" /> }`
**Scripts loaded via:** `@section Scripts { @{ await Html.RenderPartialAsync("_QuestFormScripts"); } }` — same as desktop Create.cshtml.

**Primary visual anchor:** The glass card form container. The quest title input is the topmost
interactive element below the nav bar and serves as the first visual focus point.

**Layout structure (single column, top to bottom):**
```
<div class="container-fluid px-2 mt-2">
  <div class="dm-create-card-mobile mb-3">    ← glass card; full-width
    [Page heading row]
    [Validation summary]
    <form asp-action="Create" method="post">
      [Title field]
      [Description field]
      [Challenge Rating field + help text]
      [Total Player Count field]
      [DM Session checkbox]
      [Proposed Dates section]
        [date row × N]
        [Add Another Date Option button]
      [Form action row: Create Quest + Cancel]
    </form>
  </div>
</div>
```

No sidebar. The "Quest Creation Tips" `col-lg-4` is omitted entirely (D-01).

**Page heading row:**
```html
<div class="mb-3">
    <h5 class="mb-0">
        <i class="fas fa-scroll text-warning me-2"></i>Create New Quest
    </h5>
</div>
```
Font: system sans-serif, 20px (h5), weight 700. Icon color: Bootstrap `text-warning`.

**Form field pattern (all text/number fields):**
```html
<div class="mb-3">
    <label asp-for="Title" class="form-label dm-create-label">
        Quest Title <span class="text-danger">*</span>
    </label>
    <input asp-for="Title" class="form-control" placeholder="..." />
    <span asp-validation-for="Title" class="text-danger"></span>
</div>
```
- Label: 16px, weight 400 (Bootstrap `.form-label` default).
- Input: `min-height: 44px` from mobile.css. Full width via Bootstrap `.form-control`.
- Help text (where present on desktop): `<div class="form-text">...</div>` — rendered as `<small>`,
  14px, weight 400. Omit the challenge rating "This helps players understand..." help text on mobile
  to reduce form height; the label "Challenge Rating" is self-explanatory.
- Validation span: `text-danger`, shown only on POST error.

**Textarea (Description field):**
```html
<textarea asp-for="Description" class="form-control" rows="4"
          placeholder="Describe the quest..."></textarea>
```
`rows="4"` gives adequate height without overwhelming the mobile viewport. Mobile.css `min-height: 44px`
applies but textarea natural height (4 rows ≈ 96px) overrides it.

**DM Session checkbox:**
```html
<div class="mb-3">
    <div class="form-check">
        <input asp-for="DungeonMasterSession" class="form-check-input" type="checkbox" />
        <label asp-for="DungeonMasterSession" class="form-check-label">
            <i class="fas fa-crown text-warning me-2"></i>Dungeon Master Session Only
        </label>
    </div>
    <div class="form-text">Check to make this quest visible to DMs and Admins only.</div>
</div>
```
Checkbox itself is not subject to 44px min-height (it is a small input); the label adjacent to it
provides the touch target. Bootstrap `.form-check-label` padding ensures ≥44px tap height on mobile.

**Proposed Dates section:**
```html
<div class="mb-3">
    <label class="form-label dm-create-label">
        Proposed Dates & Times <span class="text-danger">*</span>
    </label>
    <div id="proposed-dates">
        @for (int i = 0; i < Model.ProposedDates.Count; i++)
        {
            <div class="mb-3 proposed-date-item">
                <label class="form-label small">Proposed Date @(i + 1)</label>
                <div class="input-group">
                    <input asp-for="ProposedDates[i]" type="datetime-local"
                           class="form-control" required step="60" />
                    <button type="button" class="btn btn-danger"
                            onclick="removeProposedDate(this)">
                        <i class="fas fa-trash me-1"></i>Remove
                    </button>
                </div>
                <span asp-validation-for="ProposedDates[i]" class="text-danger"></span>
            </div>
        }
    </div>
    <button type="button" class="btn btn-warning btn-sm" onclick="addProposedDate()">
        <i class="fas fa-plus me-1"></i>Add Another Date Option
    </button>
    <span asp-validation-for="ProposedDates" class="text-danger d-block mt-1"></span>
</div>
```
- `input-group` layout retained (D-02). On a 375px screen the `datetime-local` input occupies ≈
  75% width and the Remove button occupies ≈ 25%. This is valid per D-02.
- `datetime-local` fires native touch pickers on iOS/Android (D-02). No custom picker JS.
- "Add Another Date Option": `btn-sm` retained from desktop. Min-height 44px from mobile.css `.btn`.
- Help text below the button section is omitted on mobile (reduces noise).

**Form action row:**
```html
<div class="d-flex gap-2 mt-3">
    <button type="submit" class="btn btn-warning flex-fill">
        <i class="fas fa-plus me-2"></i>Create Quest
    </button>
    <a href="@Url.Action("Index", "Home")" class="btn btn-secondary flex-fill">
        <i class="fas fa-times me-2"></i>Cancel
    </a>
</div>
```
Both buttons use `flex-fill` to share equal width. Min-height 44px from mobile.css. No
`justify-content-between` with only these two buttons — equal-fill is more balanced on narrow screens.

All three new CSS files follow the `dm-{view}-{element}` prefix pattern:

| File | Class prefix | Glass card class | Section heading class |
|------|-------------|------------------|-----------------------|
| dm-create.mobile.css | `dm-create-` | `.dm-create-card-mobile` | (form-label is section heading) |

### Loading Pattern

Each mobile view declares:
```cshtml
@section Styles {
    <link href="~/css/{dm-view}.mobile.css" asp-append-version="true" rel="stylesheet" />
}
```
`_Layout.Mobile.cshtml` renders `@await RenderSectionAsync("Styles", required: false)` — all three
views use this pattern (D-09).

### CSS Authoring Rules (inherited from Phase 13)

1. No `@media` queries — per-page CSS is mobile-only by definition.
2. No `@import` — do not reference `site.css`, `quests.css`, `calendar.css`, or any desktop CSS.
3. Glass card hex values are LOCKED — use exactly `rgba(255,255,255,0.15)` / `rgba(255,255,255,0.3)`
   / `rgba(0,0,0,0.2)` / `border-radius: 12px`. Do not introduce variants.
4. Parchment text values are LOCKED — use exactly `#F4E4BC` and `rgba(244,228,188,0.7)` with the
   specified text-shadow values. Do not introduce new hex colors for text.
5. Badges inside glass cards: always declare `text-shadow: none !important` to prevent parchment
   shadow from leaking onto badge text.
6. New surface colors: none. Use existing `rgba(255,255,255,0.1)` for placeholder avatar background.
